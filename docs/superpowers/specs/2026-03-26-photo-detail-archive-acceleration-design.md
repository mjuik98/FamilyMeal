# Photo Detail, Archive, and Fast Add Design

## Goal

사진 기록 앱의 핵심 경험을 강화한다. 홈은 주간 저널로 유지하고, 상세는 사진 중심으로, 작성은 최소 입력으로, 아카이브는 탐색 중심으로 재구성한다.

## Scope

### 1. 식사 상세를 사진 중심 화면으로 강화

- 상세 상단은 큰 사진 무대(hero stage)와 확대 보기 기능을 가진다.
- 같은 날짜의 다른 기록을 썸네일 스트립으로 노출한다.
- 썸네일을 누르면 현재 상세 화면 안에서 다른 기록으로 이동한다.
- 사진 아래에는 식사 메타 정보와 반응/댓글 섹션을 분리해서 배치한다.

### 2. 작성 흐름 단축

- 설명은 선택 입력으로 바꾼다.
- 설명이 비어 있을 때는 식사 종류와 인원 기준의 짧은 기본 문구를 자동 생성한다.
- 최근에 사용한 식사 종류/참여자 기본값을 계속 기억한다.
- 저장 후에는 홈으로 돌아가되 저장 날짜가 바로 선택되도록 한다.
- 빠른 저장 카피와 보조 안내를 추가해 “사진 먼저, 설명은 나중” 흐름을 강조한다.

### 3. 주간 스트립 상태 강화

- 날짜 버튼은 `기록 있음`, `기록 없음`, `오늘 미기록`, `선택됨` 상태를 시각적으로 구분한다.
- 기록이 있는 날은 썸네일과 개수 배지를 보인다.
- 오늘이면서 기록이 없으면 빈 날짜보다 더 강하게 강조한다.

### 4. 아카이브 탐색 경험 강화

- 결과를 월 단위 그룹으로 묶는다.
- 첫 진입은 일부만 노출하고 `더 보기`로 확장한다.
- 현재 결과 기준으로 “자주 함께 먹은 사람” 추천 칩을 계산해 노출한다.
- 검색/필터는 유지하되 아카이브가 단순 결과 목록이 아니라 탐색 화면으로 느껴지게 한다.

### 5. 상세 화면 컴포넌트 분리

- `MealCard`의 책임을 분해한다.
- 목표 분리는 다음과 같다.
  - `MealPhotoStage`: 사진, 확대 보기, 같은 날 썸네일 이동
  - `MealDetailSummary`: 식사 종류, 시간, 설명, 참여자, 편집/삭제
  - `MealConversationPanel`: 반응 바, 댓글 토글, 댓글 스레드, 작성
- 댓글 컴포넌트는 기존 `components/comments/*`를 계속 활용한다.

## Data and State

- 상세 화면은 현재 식사 외에 `sameDayMeals` 목록을 같이 불러온다.
- 실데이터는 `getMealsForDate`, QA는 `createQaMockMeals`를 사용한다.
- 작성 저장 후 복귀 날짜는 쿼리스트링 `?date=YYYY-MM-DD`로 전달한다.
- 홈은 이 쿼리값을 초기 선택 날짜로 사용한다.
- 아카이브는 이미 가져온 식사 목록을 월별로 그룹핑하고, 클라이언트 상태로 확장 개수를 관리한다.

## UX Decisions

- 상세에서 같은 날 이동은 뒤로가기 스택을 더럽히지 않도록 `router.replace`를 사용한다.
- 빠른 저장은 사진이 없는 경우 허용하지 않는다.
- 자동 생성 설명은 너무 기계적으로 보이지 않게 `점심 기록`, `엄마와 함께한 간식 기록` 같은 짧은 형태로 만든다.
- 아카이브 추천 칩은 검색 결과를 더 좁히는 보조 필터로만 동작하고, 선택 해제 가능해야 한다.

## Testing

- UI 정적 테스트로 새 컴포넌트 분리와 빠른 저장 로직 존재 여부를 확인한다.
- E2E로 다음 시나리오를 검증한다.
  - 상세에서 같은 날 썸네일 이동
  - 주간 스트립의 오늘 미기록 상태
  - 설명 없이 작성 후 홈 복귀
  - 아카이브의 월별 그룹과 더 보기

## Risks

- 상세에서 식사 이동과 댓글 로딩이 섞이면 깜빡임이 생길 수 있다.
- 빌드 시 `.next` 타입 산출물과 `typecheck`가 충돌한 이력이 있으므로 검증은 순차로 마무리한다.
